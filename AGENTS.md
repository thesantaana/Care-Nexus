# CareNexus 轻量版开发与过程记录规则

本文档适用于 `lite_develop` 及其派生分支。轻量版只服务管理员与护工两类角色，任何开发、测试和文档工作都必须以当前代码与数据库脚本为准，不得把 `develop` 中已经移除的完整版业务重新带回。

## 1. 项目边界

### 保留范围

- 认证、JWT、RBAC 和账号状态。
- 培训类别、标签、资源与文件存储。
- 学习访问、逐课考核、成绩汇总与错题。
- 富文本课程笔记、讨论、回复、点赞和课后作业。
- AI 培训问答、总结、学习建议、题目草稿和审核。
- 管理员工作台、护工移动端和统一门户。
- 操作日志与必要的过程审计。

### 明确排除

- 护理预约、地址、订单、分配、服务执行、评价与投诉。
- 医生授权、健康档案、健康记录、预警、随访、干预与评估。
- 老人、家属、医生、健康管理人员、运营人员和独立培训管理员角色。
- 医疗诊断、处方、治疗建议或自动医疗决策。

发现旧文档或旧代码仍描述上述排除范围时，应先判断其是否属于历史记录。现行说明应修正，历史日志不得篡改。

## 2. 分支规则

- `main`：稳定基线。
- `develop`：完整版历史开发分支。
- `lite_develop`：轻量版集成基线。
- `feature/*`：人工功能分支。
- `agent/*`：自动化整理或辅助开发分支。

轻量版 PR 默认以 `lite_develop` 为目标分支。除非任务明确要求，不得直接修改 `main`、`develop` 或 `lite_develop`。

## 3. 技术基线

- 后端：Java 8、Spring Boot 2.7.18、Spring Security、JWT、MyBatis-Plus 3.5.5。
- 数据库：MySQL 8，按 `database/init/001` 至 `008` 顺序执行。
- 管理端与护工端：Vue 3、Vue Router、Vite。
- 统一门户：React 18、TypeScript、Vite、Tiptap。
- AI：`mock` 与 `deepseek` 两种模式，通过配置切换。

不要在未形成 ADR 的情况下更换核心框架、Java 主版本、数据库或前端构建体系。

## 4. 模块边界

- `auth`：认证、Token、当前用户、账号状态和权限加载。
- `training`：资源、分类标签、学习、考核、成绩、笔记、讨论与作业。
- `ai`：培训资料 AI 适配、题目草稿生成和审核。
- `file`：文件元数据与本地存储。
- `audit`：操作日志。
- `common`：通用响应、异常、安全配置和基础设施。

跨模块调用优先依赖 Service 接口，不得直接访问其他模块的 Mapper。Controller 只负责协议转换、参数校验和调用业务服务，不堆积业务规则。

## 5. 后端实现规则

- 所有业务接口使用 `/api/v1` 前缀。
- 统一返回 `ApiResponse<T>`；分页使用 `PageResponse<T>`。
- 业务冲突使用明确错误码和 HTTP 状态，不得用 HTTP 200 包装失败。
- DTO、VO、Entity 分离；不得直接把数据库实体作为外部响应。
- 权限判断使用权限码，不在 Controller 中硬编码中文角色名。
- 用户范围数据必须从当前认证上下文获取，不接受前端传入任意用户 ID 替代。
- 状态流转应检查旧状态，必要时使用原子更新避免重复操作。
- 列表查询避免 N+1；跨表批量加载后组装。
- 新增 Service 时保持职责单一。文件过大或混合多个子领域时应拆分。

## 6. 数据库规则

当前最终结构为 28 张表。新增或修改数据库结构时：

1. 不直接改写已经用于环境初始化的历史迁移语义，优先新增顺序脚本。
2. 同步更新 `database/README.md`、数据字典和数据库设计文档。
3. 明确主键、唯一约束、索引、外键与状态字段。
4. 种子数据必须可重复执行或明确只用于全新数据库。
5. 不把数据库密码、真实手机号、密钥或其他敏感数据写入 SQL。
6. 迁移脚本必须在 MySQL 8 环境实际执行验证后才能记录为通过。

## 7. 安全规则

- 密码只保存 BCrypt 哈希。
- JWT Secret、数据库密码和 AI Key 仅通过环境变量或本地安全配置提供。
- 生产环境不得使用 README 中的演示账号和默认 Secret。
- 上传文件必须校验扩展名、MIME、大小和存储路径。
- 错误响应不得泄露堆栈、SQL、密钥、完整手机号或文件系统绝对路径。
- 操作日志不得记录密码、Token、AI Key 或完整敏感内容。

## 8. AI 规则

AI 只能处理调用者有权访问的文本培训资料。

- 普通护工仅可使用已发布资料。
- AI 回答必须保留来源资源引用。
- 资料不足时明确说明，不得编造。
- 不读取或推断健康档案、诊断、处方和个体化治疗信息。
- 题目草稿仅支持单选题和判断题。
- 审核通过只创建 `DRAFT` 正式题目，不自动发布、不自动加入考核。
- 新模型接入必须保留 `AiTrainingService` 适配边界和稳定 Mock 模式，避免测试依赖外部 API。

AI 受保护范围包括 `backend/src/main/java/com/carenexus/ai/`、相关前端页面、AI 接口和设计文档。修改时应同步检查提示词边界、数据权限和审核闭环。

## 9. 前端规则

- 不使用伪造业务记录掩盖后端接口缺失。
- 登录态、Bearer Token、401 清理、路由守卫和权限菜单保持一致。
- 管理员端只展示管理能力；护工端只展示学习相关能力。
- 新页面必须处理加载、空数据、错误、无权限和重复提交状态。
- 统一门户、管理端和护工端的路由或端口发生变化时，同步更新 README 和启动脚本。

## 10. 测试与检查

涉及对应模块时至少执行：

```text
cd backend && mvn verify
cd frontend/admin-web && npm run lint && npm run build
cd frontend/mobile-web && npm run lint && npm run build
cd frontend/portal-web && npm run build
```

数据库或真实接口发生变化时，还应执行 MySQL 8 导入与 HTTP 主链路联调。没有实际执行的检查不得写成“通过”；环境受阻应写明命令、原因和未验证范围。

## 11. 文档与历史记录

- 现行文档必须基于 `lite_develop` 当前实现，不再以完整版需求基线作为现状。
- `CHANGELOG.md`、`TEST_LOG.md`、个人日志和历史任务详情属于审计记录，原则上只追加，不回写历史结论。
- 项目状态、范围、接口、数据库、需求和设计文档在功能变化后同步更新。
- 文档不得声称未实现、未联调或未测试的功能已经完成。
- 文档中的表数量、端口、角色、接口和状态必须能从代码或 SQL 找到依据。

## 12. 提交与 PR

提交信息应简洁并体现模块，例如：

```text
feat(training): add course discussion
fix(auth): reject disabled account token
docs: refresh lite edition architecture
```

PR 说明至少包含：变更内容、原因、影响范围、验证结果和未完成事项。文档整理不得夹带无关业务代码修改。
