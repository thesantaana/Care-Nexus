# 智慧护理项目开发与过程记录规则

## 1. 项目定位

本项目为“互联网+智慧护理互联互通管理系统”。

系统包含四个逻辑业务端：

1. 护理培训系统
2. 老人/家属移动护理平台
3. 医生服务系统
4. 综合管理系统

项目采用前后端分离和模块化单体架构。

当前开发人员不负责AI功能。AI模块由其他成员独立开发。

## 2. 技术规范

后端：

* Java 8+
* Maven
* Spring
* Spring MVC
* Spring Boot
* MyBatis或MyBatisPlus
* MySQL 8.0
* Redis
* JUnit 5

前端：

* JavaScript
* Vue
* 使用仓库已有UI组件库
* PC端和移动端分开组织

工程：

* Git
* nginx
* Docker
* JMeter
* RESTful API
* UML
* CDM/PDM

实际仓库已经确定的版本优先。禁止未经批准更换框架或大版本。

## 3. 架构规则

后端采用模块化单体，不改造成微服务。

业务模块至少包括：

* common
* security
* user
* training
* care
* health
* system
* integration

模块内部按以下结构组织：

* controller
* service
* mapper
* entity
* dto
* vo
* converter
* enums

禁止：

* Controller直接操作Mapper
* Controller承载复杂业务逻辑
* 跨模块直接操作其他模块的数据表
* 循环依赖
* 超长方法
* 复制粘贴相同业务逻辑
* 将所有代码堆放在common模块

## 4. AI模块边界

AI模块由其他开发人员负责。

当前任务禁止：

* 实现AI功能
* 修改AI业务逻辑
* 修改AI接口协议
* 修改AI负责人维护的目录
* 创建模拟AI回答
* 添加Spring AI依赖
* 添加向量数据库或RAG功能

发现已有AI目录时，将其记录到：

```text
docs/management/MODULE_OWNERSHIP.md
```

并标记为受保护目录。

## 5. 代码规范

* 类和接口使用大驼峰命名。
* 变量和方法使用小驼峰命名。
* 常量使用全大写加下划线。
* 命名必须表达真实业务含义。
* 使用统一代码格式化配置。
* 删除无用导入和无用依赖。
* 类、接口和公共方法添加有效注释。
* 关键业务规则添加原因型注释。
* 禁止添加解释代码表面含义的无意义注释。
* 遵循单一职责和开闭原则。
* 避免超过三层的复杂嵌套。
* 异常必须转换为明确的业务错误或记录日志。
* 禁止空catch。
* 禁止直接打印堆栈。
* 禁止使用System.out.println作为业务日志。

## 6. 接口规范

* 接口遵循RESTful风格。
* 使用统一返回结构。
* 使用统一错误码。
* Controller只负责参数接收、校验和返回。
* 请求参数使用DTO。
* 响应参数使用VO。
* 不直接将数据库Entity返回给前端。
* 列表接口必须考虑分页。
* 状态变更接口必须校验当前状态。
* 删除、停用和驳回操作必须验证权限。
* 接口变更必须同步更新接口文档。

## 7. 数据库规范

* 表名和字段名统一使用蛇形命名。
* 每张表必须有明确主键。
* 根据实际查询设计索引。
* 禁止对高频查询长期执行全表扫描。
* 状态字段使用统一枚举或字典。
* 创建时间、更新时间和逻辑删除字段保持统一。
* 事务边界放在Service层。
* 禁止在循环中大量执行单条数据库查询。
* 数据库变更必须形成SQL脚本。
* 禁止只修改本地数据库而不提交迁移脚本。

## 8. 安全规范

* 所有外部输入必须校验。
* 密码不得明文存储。
* 敏感配置通过环境变量注入。
* 禁止在代码、Git和日志中写入真实密码、Token和密钥。
* 日志不得输出身份证号、完整手机号、密码或健康隐私。
* 文件上传必须限制格式、大小和文件名。
* 不同角色只能访问授权资源。
* 关键操作记录操作日志。

## 9. 测试规范

核心业务必须覆盖：

* 正常场景
* 异常场景
* 边界场景
* 无权限场景
* 非法状态流转场景

完成任务前必须运行仓库中可用的：

* 单元测试
* 后端构建
* 前端lint
* 前端生产构建

没有实际运行的测试不得写成“通过”。

测试失败时必须记录：

* 执行命令
* 错误摘要
* 原因
* 是否解决
* 暂时处理方式

## 10. Git规范

分支类型：

* main
* develop
* feature/*
* fix/*
* docs/*

提交信息使用：

* feat: 新增功能
* fix: 修复问题
* refactor: 重构
* test: 测试
* docs: 文档
* style: 格式调整
* chore: 工程配置

小步提交。一个提交只处理一个主要事项。

未获得用户明确要求时，不执行git push、不合并main、不删除分支、不覆盖他人改动。

## 11. 任务执行流程

每次任务开始前：

1. 阅读AGENTS.md。
2. 阅读docs/management/PROJECT_STATUS.md。
3. 阅读docs/management/TASKS.md。
4. 检查当前Git状态。
5. 确认任务所属模块和负责人。
6. 明确验收标准。
7. 确认不会修改AI受保护目录。

编码完成后：

1. 查看Git diff。
2. 运行测试和构建。
3. 检查异常和边界情况。
4. 更新任务状态。
5. 更新当天工作日志。
6. 更新测试日志。
7. 有问题时更新问题跟踪表。
8. 架构发生变化时新增ADR。
9. 更新PROJECT_STATUS.md。
10. 输出变更和验证结果。

## 13. 工作日志规则

工作日志只能记录实际发生的事项。

每次完成任务后向当天日志追加：

* 日期
* 当前阶段
* 任务编号
* 今日目标
* 实际完成内容
* 修改文件
* 关键技术点
* 执行命令
* 测试结果
* 遇到的问题
* 解决方式
* 未完成事项
* 下一步计划
* 对应Git提交

不得虚构工作时长、测试结果、会议、提交记录或已完成功能。

## 14. 完成定义

任务只有同时满足以下条件才算完成：

* 功能达到验收标准
* 代码符合架构边界
* 没有修改受保护模块
* 构建成功
* 可用测试已经执行
* 没有新增明显控制台错误
* 接口文档已同步
* 数据库脚本已同步
* 工作日志已更新
* 测试日志已更新
* 项目状态已更新
* 未完成事项已经明确记录
