# 配置管理计划

项目名称：CareNexus 颐联  
版本：轻量版 2.0  
更新时间：2026-07-15

## 1. 目的

规范 `lite_develop` 轻量版的分支、提交、评审、文档、数据库脚本、配置和交付物管理，避免将完整版已删除模块重新引入当前产品。

## 2. 配置项

| 类别 | 配置项 |
|---|---|
| 源代码 | `backend/`、`frontend/`、`scripts/` |
| 数据库 | `database/init/`、数据字典、模型成果 |
| 需求设计 | `docs/requirements/`、`docs/design/`、`docs/api/` |
| 测试 | 测试计划、测试用例、`TEST_LOG.md` |
| 过程 | `TASKS.md`、`PROJECT_STATUS.md`、`CHANGELOG.md`、会议与个人日志 |
| 规则 | `README.md`、`AGENTS.md`、ADR |
| 配置 | `application.yml`、前端 Vite/ESLint/TypeScript 配置、环境变量样例 |
| 交付物 | 用户手册、部署说明、最终报告、CDM/PDM、答辩材料、版本标签 |

## 3. 分支策略

| 分支 | 用途 |
|---|---|
| `main` | 稳定仓库基线 |
| `develop` | 完整版历史开发线，不与轻量版整体互并 |
| `lite_develop` | 轻量版集成与验收基线 |
| `feature/*` | 人工功能开发 |
| `fix/*` | 缺陷修复 |
| `docs/*` 或 `agent/*` | 文档与自动化整理 |

所有轻量版 PR 默认合并到 `lite_develop`。从 `develop` 复用内容时只允许挑选经过范围审查的单独提交。

## 4. 提交规范

推荐格式：

```text
feat(training): add course assignment
fix(ai): reject inaccessible source
docs: refresh lite API inventory
test(auth): cover disabled account token
```

每次提交应保持单一目的，不混入无关格式化、生成文件或个人配置。

## 5. PR 规则

PR 至少说明：

- 变更内容和原因。
- 影响的角色、模块、接口和表。
- 是否改变权限、状态流转或数据结构。
- 已执行的测试与结果。
- 未验证事项。
- 相关文档是否同步。

评审重点：轻量版范围、越权、敏感配置、数据库兼容、前后端一致性和测试真实性。

## 6. 文档基线

现行基线文档：

- `README.md`
- `docs/management/LITE_SCOPE.md`
- `docs/management/PROJECT_STATUS.md`
- `docs/requirements/`
- `docs/design/`
- `docs/api/`
- `database/README.md`
- `docs/test/测试计划.md`

历史过程文档：

- `CHANGELOG.md`

历史文档原则上只追加。现行说明发生变化时直接更新基线文档，同时在 Changelog 追加变更摘要。

## 7. 数据库变更管理

- 以 MySQL 8 为基线。
- 当前脚本为 001–008，最终 28 张表。
- 新增结构优先创建后续编号迁移脚本。
- 修改数据库必须同步 Entity、Mapper、数据字典、数据库设计、API 和测试。
- 全新数据库导入通过后才能更新状态为完成。
- 不提交真实数据库密码、敏感数据和本地绝对路径。

## 8. 依赖与锁文件

- Maven 依赖只在 `backend/pom.xml` 管理。
- 三个前端各自维护 `package.json` 和 `package-lock.json`。
- 更新依赖后执行对应测试、lint/build 和生产依赖审计。
- 不提交 `node_modules/`、`target/`、`dist/`、上传目录或 IDE 临时文件。

## 9. 环境配置

以下值只通过环境变量或安全部署配置提供：

- `DB_URL`
- `DB_USERNAME`
- `DB_PASSWORD`
- `CARE_NEXUS_JWT_SECRET`
- `CARE_NEXUS_JWT_EXPIRATION_SECONDS`
- `FILE_STORAGE_ROOT`
- `AI_MODE`
- `DEEPSEEK_API_KEY`
- `DEEPSEEK_BASE_URL`
- `DEEPSEEK_MODEL`

仓库可提供变量名和安全占位值，不保存真实凭证。

## 10. 版本与标签

建议：

- 开发里程碑：`lite-v0.x`。
- 答辩候选：`lite-v1.0.0-rc.1`。
- 最终交付：`lite-v1.0.0`。

发布标签前必须：

1. 固定提交 SHA。
2. 后端 verify 通过。
3. 三个前端构建通过。
4. 干净数据库导入通过。
5. 核心 HTTP 回归通过。
6. 文档、用户手册和交付清单完成。

## 11. 备份与归档

最终交付归档：

- 源代码和标签。
- 001–008 SQL。
- 环境变量说明，不含密钥。
- 测试报告和日志。
- 用户手册与部署说明。
- CDM/PDM 与图片。
- 答辩材料和版本发布记录。

## 12. 变更控制

涉及以下内容必须由项目负责人确认：

- 恢复护理订单或医生健康管理。
- 新增角色。
- 切换核心框架或数据库。
- 引入医疗 AI。
- 修改一课一考核、60 分通过或 AI 审核边界。
- 破坏性 API 或数据库变更。

确认后先更新范围或 ADR，再实施代码。
