# 配置管理计划

## 1. 文档目的与适用范围

本文档用于规范“互联网+智慧护理互联互通管理系统”的配置管理活动，明确代码、文档、需求、接口、数据库脚本、发布包和交付物的版本控制方式。

本文档适用于本项目全体成员，包括：

- 项目计划、需求、设计、测试、交付等过程文档。
- 后续创建的后端、前端、数据库、部署和测试工程。
- Git 分支、提交、PR、评审、合并、标签和基线管理。
- AI 模块及其他成员负责模块的保护和变更边界。

当前项目尚未进入需求基线、架构设计和业务编码阶段。本文档先作为先启阶段配置管理计划使用，后续如流程发生变化，应按变更控制流程修订。

## 2. 配置管理角色与职责

| 角色 | 人员 | 主要职责 |
|---|---|---|
| 项目负责人 | 隋咏轩 | 维护主干稳定性，审核并合并 `main`，组织基线评审，确认关键变更 |
| 主要开发成员 | 隋咏轩、孙洋 | 负责主要开发任务，参与 `develop` 分支审核和日常集成 |
| 文档与需求成员 | 李亦航 | 负责需求、计划、过程文档等材料编写，提交 PR 后由项目负责人审核 |
| 测试与交付成员 | 张远航 | 负责测试计划、测试记录、交付材料等，提交 PR 后由项目负责人审核 |
| AI 模块负责人 | 待确认 | 仅在需求确认和授权后负责 AI 模块相关开发和维护 |

成员不得越权修改他人负责模块。确需跨模块修改时，必须在任务说明、PR 描述或变更记录中说明原因，并经过对应负责人和项目负责人确认。

## 3. Git 分支管理策略

项目规划使用以下分支类型：

| 分支类型 | 职责 |
|---|---|
| `main` | 稳定版本、阶段基线和发布版本，不作为日常开发分支 |
| `develop` | 需求基线确认后创建，用于日常集成和阶段内持续开发 |
| `feature/*` | 功能开发分支，用于实现已确认任务 |
| `fix/*` | 缺陷修复分支，用于修复已确认问题 |
| `docs/*` | 文档任务分支，用于计划、需求、设计、测试、管理和交付文档修改 |

`main` 必须保持可追溯、可评审、可作为阶段基线的状态。任何成员不得直接向 `main` 提交。

## 4. 分支创建时机与命名规则

### 4.1 develop 创建前

在 T-009 需求基线确认并创建 `develop` 前：

- `docs/*` 分支从 `main` 创建。
- 文档修改通过 PR 合并回 `main`。
- 不允许成员直接向 `main` 提交。
- `main` 只由隋咏轩审核并合并。

当前阶段以文档和过程管理为主，不创建前端、后端或数据库工程。

### 4.2 develop 创建后

T-009 需求基线确认并创建 `develop` 后：

- `feature/*`、`fix/*`、`docs/*` 默认从 `develop` 创建。
- 日常修改通过 PR 合并回 `develop`。
- 阶段基线和发布版本经过评审后进入 `main`。
- `main` 不作为日常开发分支。

### 4.3 分支命名规则

分支名称必须包含任务编号，格式如下：

```text
docs/T-xxx-short-name
feature/T-xxx-short-name
fix/T-xxx-short-name
```

示例：

```text
docs/T-004-config-plan
docs/T-006-requirements-spec
feature/T-012-user-login
fix/T-023-order-status
```

命名要求：

- 任务编号必须与 `docs/management/TASKS.md` 保持一致。
- `short-name` 使用英文小写单词和连字符。
- 不使用空格、中文、特殊符号或无任务编号的临时分支名。
- 一个分支只处理一个主要任务。

## 5. 提交信息规范

提交信息使用以下前缀：

| 前缀 | 含义 | 示例 |
|---|---|---|
| `feat` | 新增功能 | `feat: add user login endpoint` |
| `fix` | 修复问题 | `fix: correct order status transition` |
| `docs` | 文档修改 | `docs: add configuration management plan` |
| `test` | 测试相关 | `test: add permission test cases` |
| `refactor` | 重构 | `refactor: simplify user service` |
| `style` | 格式调整 | `style: format markdown tables` |
| `chore` | 工程配置或杂项 | `chore: update gitignore` |

提交要求：

- 小步提交，一个提交只处理一个主要事项。
- 提交内容必须真实反映已完成修改。
- 不得提交真实密码、Token、密钥、隐私数据或本地临时文件。
- 未执行的测试、构建或检查不得写成“通过”。

## 6. PR、Code Review 与合并规则

所有成员修改项目文件应通过 PR 完成。PR 应至少说明：

- 关联任务编号。
- 修改内容摘要。
- 影响范围。
- 执行过的检查、测试或未执行原因。
- 是否涉及需求、接口、数据库、权限、AI 模块或受保护模块。

评审要求：

- 文档类 PR 重点检查内容真实性、目录位置、路径引用、过程留痕和格式。
- 功能类 PR 重点检查需求符合性、分层结构、接口约定、异常处理、权限校验、测试和日志。
- 数据库类 PR 重点检查表结构、字段含义、索引、迁移脚本和兼容性。
- AI 相关 PR 必须确认当前任务属于 AI 授权任务，并由 AI 负责人或项目负责人审核。

合并要求：

- PR 未通过审核不得合并。
- 不得为了赶进度绕过评审直接提交到 `main`。
- 评审发现问题时，应修改后再次提交并等待复核。
- 合并前应确认目标分支正确。

## 7. 审核与合并权限

| 目标分支 | 审核与合并规则 |
|---|---|
| `main` | 只允许隋咏轩审核并合并 |
| `develop` | 可由隋咏轩和孙洋审核并合并 |
| 功能、修复、文档分支 | 成员可在自己分支上提交；合并必须走 PR |

李亦航和张远航主要负责文档、测试和交付材料，提交后由隋咏轩审核。涉及代码、接口、数据库或受保护模块的修改，必须先获得任务授权。

## 8. 配置项识别与目录管理

项目配置项包括但不限于：

| 配置项 | 存放位置 |
|---|---|
| 项目计划、进度计划、配置管理计划 | `docs/plans/` |
| 需求规约、用例模型、业务流程、MVP 边界 | `docs/requirements/` |
| 架构设计、概要设计、详细设计、数据库设计、原型 | `docs/design/` |
| 接口设计和接口变更记录 | `docs/api/` |
| 测试计划、测试用例、测试日志、最终测试报告 | `docs/test/` |
| 个人工作日志 | `docs/logs/` |
| 会议记录 | `docs/meetings/` |
| 项目状态、任务、问题、模块归属、变更记录 | `docs/management/` |
| 架构决策记录 | `docs/decisions/` |
| 部署说明、用户手册、产品说明、答辩材料、交付清单 | `docs/delivery/` |

不得创建重复的阶段目录来存放同类文件。RUP 阶段与目录的映射以 `docs/STAGE_INDEX.md` 为准。

## 9. 需求、架构、数据库与发布基线管理

### 9.1 需求基线

需求基线在 T-009 完成评审后形成。基线内容至少包括：

- 软件需求规约。
- 用例模型和业务流程。
- MVP 边界。
- 测试计划初稿。
- 需求评审结论。

需求基线确认前，需求文档可以修订，但必须记录实际修改内容。需求基线确认后，需求变更必须走变更控制流程。

### 9.2 架构基线

架构基线在系统架构设计说明书评审通过后形成。正式基线形成前，模块化单体、模块列表和包结构均属于候选方案。

架构基线确认后，影响模块划分、技术栈、接口协议、部署方式或数据库结构的修改，必须经过评审。

### 9.3 数据库基线

数据库基线在数据库设计和初始化脚本评审通过后形成。数据库基线形成后：

- 表结构变更必须提供 SQL 脚本。
- 字段含义、索引、状态值和兼容性影响必须记录。
- 禁止只修改本地数据库而不提交迁移脚本。

### 9.4 发布基线

发布基线在阶段评审或最终交付前形成。发布基线应包含：

- 已评审的源码和文档。
- 测试结果和缺陷状态。
- 部署说明和交付清单。
- 对应 Git 标签。

## 10. 标签与版本号规则

阶段标签暂定为：

```text
inception-v1.0
elaboration-v1.0
construction-v1.0
v1.0.0
```

标签含义：

| 标签 | 含义 |
|---|---|
| `inception-v1.0` | 先启阶段基线 |
| `elaboration-v1.0` | 精化阶段基线 |
| `construction-v1.0` | 构建阶段基线 |
| `v1.0.0` | 最终发布版本 |

标签只能在评审通过后创建。未经评审的临时版本不得标记为阶段基线或发布版本。

## 11. 需求、接口、数据库变更控制流程

需求基线确认前：

- 允许在对应文档中修订需求、接口设想和数据库设想。
- 修改应记录在相关文档、`CHANGELOG.md`、任务记录或工作日志中。
- 不得声称未评审内容已经形成基线。

需求基线确认后：

1. 提出变更原因和影响范围。
2. 记录关联任务或问题编号。
3. 分析对需求、接口、数据库、测试、进度和交付物的影响。
4. 由项目负责人组织确认。
5. 经批准后在对应分支实施。
6. 同步更新相关文档、脚本、测试和日志。
7. 通过 PR 评审后合并。

未经确认的变更不得直接修改接口协议、数据库结构、权限规则或关键状态流转。

## 12. 冲突处理与回滚策略

冲突处理原则：

- 优先保护 `main` 和阶段基线。
- 冲突由分支负责人先处理，必要时请项目负责人协调。
- 涉及需求、接口、数据库或权限冲突时，必须先确认业务规则，再处理代码或文档。
- 不得通过覆盖他人修改来解决冲突。

回滚策略：

- 文档错误优先通过新提交修正。
- 未合并分支的问题在原分支修复。
- 已合并到 `develop` 的问题通过 `fix/*` 分支修复。
- 已进入 `main` 或已打标签的基线问题，必须记录原因、影响范围和处理方式。
- 禁止未经确认执行破坏性历史重写。

## 13. 发布包与交付物归档规则

发布包和交付物统一归档到 `docs/delivery/`，测试报告统一存放在 `docs/test/`。

交付物至少包括：

- 部署说明。
- 用户手册。
- 产品说明。
- 答辩材料。
- 交付清单。

交付清单可以引用 `docs/test/` 中的测试报告，不得复制出多个不同版本的测试报告。

最终发布前应确认：

- 发布内容与 Git 标签一致。
- 文档、源码、测试记录和交付清单相互对应。
- 未包含真实密码、Token、密钥、隐私数据或本地临时文件。

## 14. AI 模块与成员模块保护规则

AI 模块负责人当前为“待确认”。在负责人和任务未确认前：

- 非 AI 负责人不得修改 AI 模块。
- 非 AI 任务不得修改 AI 业务逻辑、接口协议或依赖。
- 不得创建模拟 AI 回答。
- 不得新增 Spring AI、向量数据库或 RAG 相关依赖。

AI 负责人在完成需求确认并获得授权后，可以在已登记的 AI 受保护范围内开发 AI 功能。AI 模块范围和负责人应记录在 `docs/management/MODULE_OWNERSHIP.md`。

其他成员模块也应遵循负责人边界。跨模块修改必须说明原因，并经过相关负责人确认。

## 15. 配置管理检查与过程记录

每个任务完成前，应根据任务性质执行适用检查：

- 查看 Git 状态。
- 查看 Git diff。
- 检查文件路径和目录位置。
- 检查是否修改受保护模块。
- 检查是否涉及需求、接口、数据库或权限变更。
- 运行仓库中可用的测试、构建或 lint 命令；不适用时说明原因。
- 更新任务状态、项目状态、变更记录和工作日志。

纯文档任务不要求执行代码构建或软件测试，但必须记录未执行原因，不得写成测试通过。
