## 1. 评审目的

本文档用于对先启阶段已经完成的需求工件进行统一评审，检查软件需求规约、用例模型、业务流程和测试计划之间的一致性、可追踪性和后续设计输入完整性。

本轮评审目标是确认先启阶段需求基线。需求基线确认提交推送到 `main` 后，将创建 `inception-v1.0` 标签并从同一基线提交创建 `develop` 分支。

## 2. 评审范围

本轮评审范围包括：

- 四端 MVP 范围是否一致。
- 非 MVP 边界是否一致。
- 角色、权限和数据隔离是否一致。
- 核心业务流程是否与需求规约一致。
- 核心状态流转是否一致。
- AI 是否仅用于护理培训辅助。
- 医生端是否保持轻量健康管理定位。
- 需求、用例、业务流程和测试范围是否可追踪。
- 后续 T-010 详细测试用例设计和 T-011 架构、数据库设计是否具备明确输入。

本轮不评审接口路径、数据库表结构、页面原型、工程目录和具体代码实现。

## 5. 四端 MVP 范围确认

| 业务端 | MVP 范围 | 评审结论 |
|---|---|---|
| 护理培训系统 | 培训类别、标签、文章、视频、PPT、学习记录、考核、AI 辅助培训 | 范围一致，是最高优先级和主选题 |
| 老人/家属移动护理平台 | 注册登录、家属绑定老人、服务浏览、地址管理、预约下单、订单取消、评价和投诉 | 范围一致，以服务闭环为目标 |
| 护工/护理人员服务执行 | 查看分配订单、确认服务、开始服务、完成服务、填写服务记录 | 范围一致，与订单执行流程对应 |
| 医生服务系统 | 授权老人健康档案、健康记录、健康预警、随访、干预、健康评估 | 范围一致，保持轻量健康管理定位 |
| 综合管理系统 | 用户、角色、权限、账号状态、服务项目、订单分配、培训内容、医生老人授权、基础数据、操作日志 | 范围一致，支撑跨端管理和权限治理 |

## 6. 非 MVP 范围确认

以下内容在软件需求规约、业务流程和测试计划中保持一致，均不纳入首期：

- 医生端 AI 诊断。
- 医生端 AI 处方。
- 医生端自动医疗决策。
- 老人端 AI 问诊。
- 管理端 AI 决策。
- 真实支付。
- 真实短信。
- HIS/EMR 真实对接。
- 可穿戴设备实时接入。
- 直播培训。
- 线下签到。
- 证书自动发放。
- 智能派单。
- 复杂多角色切换。
- 完整高并发压测。

评审结论：非 MVP 边界明确，不阻塞需求基线草案。

## 7. 角色和权限确认

| 角色 | 权限边界 | 评审结论 |
|---|---|---|
| 管理员 | 用户、角色、权限、账号状态、基础数据、操作日志、医生老人授权 | 一致 |
| 运营人员 | 服务项目、订单人工分配、投诉处理 | 一致 |
| 培训管理员 | 培训类别、标签、文章、视频、PPT、考核、AI 题目草稿审核 | 一致 |
| 护工/护理人员 | 学习培训资源、参加考核、查看学习记录、执行分配给自己的订单 | 一致 |
| 老人 | 浏览服务、地址、预约、订单、评价、投诉 | 一致 |
| 家属 | 绑定老人，代老人浏览服务、预约、查看订单、评价和投诉 | 一致 |
| 医生/健康管理人员 | 使用已有授权查看和维护老人健康数据；健康管理人员可参与授权维护 | 一致 |
| AI 服务 | 仅作为护理培训辅助外部服务 | 一致 |

权限和数据隔离重点包括：

- 医生只能访问授权老人。
- 家属只能访问已绑定老人相关数据。
- 护工/护理人员只能访问分配给自己的订单。
- 不同业务角色只能访问授权功能和授权数据。
- AI 模块修改边界属于配置管理和过程审计检查，不属于系统运行时权限测试。

## 9. AI 边界确认

AI 首期仅用于护理培训辅助，范围包括：

- 培训资料问答。
- 知识点总结。
- 学习建议。
- 基于已上传或已入库培训资料生成题目、选项、答案、解析和练习内容草稿。

AI 题目草稿必须经过培训管理员审核、修改和发布后才能进入正式题库或考核。

AI 不进入医生端诊断、处方、自动医疗决策、老人端问诊或管理端决策。

AI 模块代码、接口协议、依赖和配置的变更应符合 `docs/management/MODULE_OWNERSHIP.md`。该项属于仓库过程检查，不作为系统运行时权限测试。

以下事项需要在精化阶段或详细设计阶段继续细化，但不阻塞当前需求基线 v1.0：

- 老人/家属绑定采用老人账号确认、绑定码或身份验证中的哪一种具体交互方式。

- 文件上传格式、大小、存储策略和预览方式。
- 具体接口路径、请求字段、响应字段和错误码。
- 具体数据库表结构、字段类型、索引和约束。
- 具体页面路由、菜单位置、原型布局和交互细节。
- 测试环境具体命令、详细测试用例步骤和测试数据脚本。

## 14. 基线后的变更控制规则

需求基线正式确认后：

1. 影响需求范围、核心流程、权限、状态流转、接口协议、数据库结构或 AI 边界的修改必须走变更控制流程。
2. 变更应说明原因、影响范围、关联需求编号、关联用例编号、测试影响和负责人。
3. 变更需由项目负责人确认后才能实施。
4. 变更实施后必须同步更新需求规约、用例模型、业务流程、测试计划或后续详细设计工件。
5. 未经确认不得直接修改核心业务规则、订单状态、医生授权、AI 边界或非 MVP 范围。
