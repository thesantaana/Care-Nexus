# 需求澄清与确认流程

更新时间：2026-07-15  
适用分支：`lite_develop`

## 1. 目的

确保新增需求、缺陷修复和文档调整不会突破 CareNexus 轻量版边界，也不会在信息不足时凭空假设接口、数据库或业务规则。

## 2. 先核对现状

处理任务前按优先级查看：

1. 用户本轮明确要求。
2. `docs/management/LITE_SCOPE.md`。
3. `docs/management/PROJECT_STATUS.md`。
4. 当前分支代码、路由、SQL 和测试。
5. 现行需求、设计和 API 文档。
6. 历史 Changelog、Test Log 和任务记录。

历史记录可以解释项目过程，但不能覆盖当前代码事实。

## 3. 范围判断

### 可直接进入方案的轻量版需求

- 管理员、护工账号和权限。
- 培训资源、分类标签、学习、考核、成绩和错题。
- 笔记、讨论、点赞和作业。
- AI 培训问答、总结、建议和题目草稿审核。
- 文件、审计、门户、管理端和护工端。

### 必须先升级为范围变更的需求

- 护理预约订单、老人家属、医生健康管理。
- 新增业务角色。
- 医疗诊断、处方、治疗或自动决策。
- 更换 Java/Spring、数据库或主要前端框架。
- 大规模微服务化。
- 破坏性 API 或数据库变更。

范围变更先更新 `LITE_SCOPE.md` 和 ADR，经项目负责人确认后再开发。

## 4. 澄清要点

当需求存在实质歧义时，至少确认：

- 使用角色和数据范围。
- 前端入口与页面。
- 接口路径、请求和响应。
- 数据库是否新增表/字段/状态。
- 状态流转和重复操作规则。
- 权限码和本人数据限制。
- 文件类型、大小和存储方式。
- AI 来源、模型、审核和安全边界。
- 验收场景和实际测试要求。

对不影响核心实现的小细节，可依据当前项目约定作合理选择并在 PR 中说明，不要用无关问题阻断任务。

## 5. 方案输出

方案应明确：

1. 变更文件与模块。
2. 前后端和数据库影响。
3. 兼容性与迁移方式。
4. 安全与权限风险。
5. 测试命令和主链路。
6. 需要同步的文档。
7. 不在本次处理的事项。

## 6. 实施规则

- 从 `lite_develop` 创建 feature/fix/docs/agent 分支。
- 不直接恢复 `develop` 中已删除的 care/doctor 代码。
- Controller、Service、Mapper、DTO/VO 分层。
- 数据库变化使用顺序脚本。
- AI 使用现有适配层，不绕过来源权限和人工审核。
- 不把密钥、密码、Token 或本地路径提交到仓库。

## 7. 验证规则

按影响范围执行：

```text
cd backend && mvn verify
cd frontend/admin-web && npm run lint && npm run build
cd frontend/mobile-web && npm run lint && npm run build
cd frontend/portal-web && npm run build
```

数据库变化还需干净 MySQL 8 导入；接口变化需真实 HTTP 联调。未执行的检查必须明确写为未执行。

## 8. 文档同步

以下内容变化时必须同步：

- 角色/范围：LITE_SCOPE、需求、用例、流程。
- 页面/模块：原型、架构、README。
- 接口：API 规范、接口清单、测试用例。
- 表/字段：SQL、数据字典、数据库设计、模型说明。
- 状态/权限：需求、设计、API 和测试。
- AI：架构、接口、安全边界和测试。

## 9. 历史记录

`CHANGELOG.md`、`TEST_LOG.md`、个人日志和历史任务原则上只追加。发现历史内容与当前状态不同，应在现行文档中说明版本差异，不回写历史结论。

## 10. 完成标准

任务只有在以下条件满足时才能标记完成：

- 实现与轻量版范围一致。
- 代码、数据库、前端和文档互相一致。
- 权限和本人数据边界经过验证。
- 对应构建或测试实际执行并记录。
- 未完成和已知限制明确。
- PR 说明完整，目标分支为 `lite_develop`。
