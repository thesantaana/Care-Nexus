# 软件需求规约

项目名称：CareNexus 颐联  
版本：轻量版 2.0  
更新时间：2026-07-15  
基线分支：`lite_develop`

## 1. 引言

CareNexus 轻量版是一套面向养老护理人员的培训、考核与 AI 辅助学习平台。系统由统一门户、管理员工作台、护工学习端、后端服务和 MySQL 数据库组成。

本规约只描述当前轻量版。护理预约订单、老人家属服务、医生健康管理以及医疗决策均不在范围内。

## 2. 用户角色

### 2.1 管理员 `ADMIN`

负责培训内容和学习结果管理：

- 登录并进入管理员工作台。
- 管理培训类别、标签和资源。
- 管理题目、考核和课程关联。
- 查看护工逐课成绩和整体培训状态。
- 生成、查看和审核 AI 题目草稿。
- 查看必要的操作结果与系统状态。

### 2.2 护工 `CAREGIVER`

负责完成培训学习：

- 登录并进入护工学习端。
- 浏览已发布课程和课程详情。
- 记录学习访问与学习时长。
- 参加每门课程的考核并查看成绩、解析和错题。
- 查看个人课程成绩和整体培训状态。
- 编辑个人富文本课程笔记。
- 参与课程讨论、回复和点赞。
- 查看课后作业并提交答案。
- 使用资料问答、总结和学习建议等 AI 辅助能力。

## 3. 总体业务规则

1. 系统仅保留管理员与护工两类业务角色。
2. 护工只能访问已发布培训资源。
3. 培训资源状态为 `DRAFT`、`PUBLISHED`、`OFFLINE`。
4. 每门已发布课程绑定一份考核。
5. 题型仅支持 `SINGLE_CHOICE` 和 `TRUE_FALSE`。
6. 考核客观题自动评分，默认 60 分通过。
7. 个人课程成绩以考试记录计算最高分、平均分和通过状态。
8. 全部课程通过后，整体培训状态为通过。
9. 个人笔记按用户和课程隔离。
10. AI 只能使用调用者有权访问的文本培训资料。
11. AI 题目草稿必须由管理员审核；通过后只创建 `DRAFT` 正式题目。
12. 系统不提供医疗诊断、处方、个体化治疗或自动医疗决策。

## 4. 功能需求

### 4.1 认证与权限

| 编号 | 需求 |
|---|---|
| REQ-AUTH-01 | 用户可使用用户名和密码登录，成功后获得 Bearer Token |
| REQ-AUTH-02 | 系统可返回当前用户、主角色、账号状态和权限码 |
| REQ-AUTH-03 | 用户可退出登录，退出后的 Token 不可继续使用 |
| REQ-AUTH-04 | 停用、逻辑删除、错误密码和无效 Token 必须被拒绝 |
| REQ-AUTH-05 | 后端根据权限码校验接口访问，前端根据角色和权限控制路由与菜单 |

### 4.2 培训分类与标签

| 编号 | 需求 |
|---|---|
| REQ-CAT-01 | 管理员可新增、修改、启用和停用培训类别 |
| REQ-CAT-02 | 管理员可新增、修改、启用和停用培训标签 |
| REQ-CAT-03 | 护工查询时只返回可用于已发布课程的有效分类信息 |

### 4.3 培训资源

| 编号 | 需求 |
|---|---|
| REQ-RES-01 | 支持文章、视频和 PPT 三类资源 |
| REQ-RES-02 | 支持文本、本地文件和外部链接三种存储方式 |
| REQ-RES-03 | 管理员可创建、修改、分页查询和查看资源详情 |
| REQ-RES-04 | 管理员可发布草稿、下架已发布资源和重新发布下架资源 |
| REQ-RES-05 | 已发布资源修改前必须先下架，重复发布或下架应返回冲突 |
| REQ-RES-06 | 支持课程封面、摘要、学习时长、类别和标签 |
| REQ-RES-07 | 护工只能查询和查看已发布资源 |

### 4.4 学习记录

| 编号 | 需求 |
|---|---|
| REQ-LEARN-01 | 护工访问课程时可记录本次学习秒数 |
| REQ-LEARN-02 | 系统累计个人总学习时长和最近学习时间 |
| REQ-LEARN-03 | 系统返回个人课程访问、考试与通过状态 |
| REQ-LEARN-04 | 学习记录只能由本人读取，不能由前端指定其他用户 |

### 4.5 题库与考核

| 编号 | 需求 |
|---|---|
| REQ-EXAM-01 | 管理员可创建和修改单选题、判断题及其选项 |
| REQ-EXAM-02 | 管理员可创建、修改、发布考核并替换考核题目 |
| REQ-EXAM-03 | 每门课程最多有一份有效考核 |
| REQ-EXAM-04 | 护工可获取已发布考核并提交答案 |
| REQ-EXAM-05 | 系统自动评分并保存考试次数、答案、分数和通过状态 |
| REQ-EXAM-06 | 护工可查看个人逐课成绩、最高分、平均分和错题 |
| REQ-EXAM-07 | 管理员可查看护工培训成绩汇总 |

### 4.6 富文本笔记

| 编号 | 需求 |
|---|---|
| REQ-NOTE-01 | 护工可按课程读取和保存个人笔记 |
| REQ-NOTE-02 | 笔记支持标题、基础格式、列表、待办、引用、代码、对齐、链接、图片和表格 |
| REQ-NOTE-03 | 支持笔记图片上传和访问 |
| REQ-NOTE-04 | 用户不能读取或修改其他用户的笔记 |

### 4.7 课程讨论

| 编号 | 需求 |
|---|---|
| REQ-DISC-01 | 用户可查看课程讨论并按指定顺序排序 |
| REQ-DISC-02 | 用户可创建讨论主题和回复 |
| REQ-DISC-03 | 回复支持父回复关系 |
| REQ-DISC-04 | 用户可点赞或取消点赞主题与回复 |
| REQ-DISC-05 | 用户只能删除本人创建的主题或回复 |

### 4.8 课后作业

| 编号 | 需求 |
|---|---|
| REQ-ASG-01 | 护工可查看课程已发布作业 |
| REQ-ASG-02 | 护工可提交个人作业答案 |
| REQ-ASG-03 | 同一用户对同一作业只保留一个有效提交关系 |

### 4.9 AI 培训辅助

| 编号 | 需求 |
|---|---|
| REQ-AI-01 | 用户可基于有权访问的文本资料进行问答 |
| REQ-AI-02 | 用户可生成资料总结和个人学习建议 |
| REQ-AI-03 | 管理员可生成单选题或判断题草稿 |
| REQ-AI-04 | 管理员可分页查看草稿、查看详情并审核通过或驳回 |
| REQ-AI-05 | AI 响应保留来源资源引用 |
| REQ-AI-06 | 审核通过只创建题库草稿，不自动发布或加入考核 |
| REQ-AI-07 | 默认 Mock 模式可稳定运行，可配置 DeepSeek 模式 |
| REQ-AI-08 | 资料不足时明确说明，不得编造或输出医疗决策 |

## 5. 非功能需求

### 5.1 安全性

- 密码使用 BCrypt 哈希保存。
- JWT Secret、数据库密码和 AI Key 不得写入仓库。
- 未认证返回 401，无权限返回 403。
- 文件上传校验大小、扩展名、MIME 和存储路径。
- 日志不得记录密码、Token、密钥和完整敏感信息。

### 5.2 可靠性

- 状态流转拒绝重复或非法操作。
- 考试提交、审核和点赞等操作应避免重复数据。
- 接口异常使用统一错误结构，不向客户端暴露堆栈。
- 默认 Mock AI 不依赖外部网络，保证开发和测试稳定性。

### 5.3 性能

- 分页接口默认每页 10 条，最大 100 条。
- 列表组装避免逐条查询题目、选项、标签或成绩。
- 文件与视频大小按后端配置限制。

### 5.4 易用性

- 管理端和护工端提供加载、空数据、错误和无权限状态。
- 护工学习页面适配移动端。
- 富文本笔记提供固定工具栏和文档式编辑区域。
- 统一门户清晰区分管理员和护工入口。

### 5.5 可维护性

- 后端按 `auth`、`training`、`ai`、`file`、`audit` 和 `common` 分模块。
- Controller、Service、Mapper、DTO、VO 和 Entity 职责分离。
- 数据库变更使用顺序脚本并同步数据字典。
- 功能变化同步更新现行需求、接口、设计和测试文档。

## 6. 系统外部边界

### 包含

- MySQL 8 数据库。
- 本地文件存储。
- 可选 DeepSeek API。
- 浏览器访问的门户、管理端和护工端。

### 不包含

- 微信小程序、原生 App。
- Redis、消息队列或对象存储的强制依赖。
- 正式 Docker/Kubernetes 部署。
- 医疗机构 HIS/EMR 接口。
- 健康监测设备接入。

## 7. 验收主线

1. 管理员登录并创建、发布培训课程。
2. 管理员维护题目和逐课考核。
3. 护工登录、学习课程并记录学习情况。
4. 护工参加考核，系统自动评分并更新成绩。
5. 护工维护笔记、讨论并提交作业。
6. 护工使用 AI 问答、总结和建议。
7. 管理员生成并审核 AI 题目草稿。
8. 管理员查看护工逐课成绩和整体培训状态。

## 8. 追踪与变更

需求变更应同步更新用例、业务流程、API 清单、数据库设计和测试用例。历史完整版需求基线仅作为过程记录，当前验收以本规约和 `lite_develop` 实现为准。
