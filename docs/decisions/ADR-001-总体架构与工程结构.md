# ADR-001 轻量版总体架构与工程结构

日期：2026-07-15  
状态：已接受  
适用分支：`lite_develop`

## 1. 背景

CareNexus 早期完整版同时规划护理培训、护理预约订单、护工服务执行、医生健康管理和综合管理。为适应当前实训周期、交付风险和实际开发进度，项目在 2026-07-11 从 `develop` 派生 `lite_develop`，将产品收敛为管理员与护工使用的培训平台。

旧 ADR 中的多角色、care/doctor 模块和 36 表结构不再适用于轻量版，需要重新确认现行架构决策。

## 2. 决策

### 2.1 产品范围

轻量版只保留：

- 管理员和护工两类角色。
- 培训资源、学习、逐课考核、成绩、错题。
- 富文本笔记、课程讨论、点赞和课后作业。
- AI 培训问答、总结、建议、题目草稿和审核。
- 文件存储、认证权限和操作审计。

删除护理订单、老人家属、医生健康管理和医疗决策能力。

### 2.2 架构形态

采用模块化单体后端，而不是微服务：

- 单一 Spring Boot 应用。
- 单一 MySQL 数据库。
- 后端按 `auth`、`training`、`ai`、`file`、`audit`、`common` 分模块。
- 模块间通过 Service 协作，不直接跨模块访问 Mapper。

原因：当前团队规模、交付周期和业务复杂度不足以支撑微服务带来的部署、链路追踪和一致性成本。

### 2.3 前端拆分

保留三个前端工程：

- React + TypeScript 统一门户。
- Vue 管理员端。
- Vue 护工移动端。

原因：现有代码已经形成三个独立入口，分别承担品牌门户、管理工作台和移动学习体验。当前不进行大规模框架合并，以降低重构风险。

### 2.4 数据库

使用 MySQL 8 和顺序 SQL 脚本。最终结构为 28 张表。

- 不使用早期完整版 36 表模型。
- 数据库变更按编号脚本追加。
- PowerDesigner 最终模型从当前 SQL 生成。

### 2.5 AI 适配

使用 `AiTrainingService` 适配接口：

- 默认 Mock，保证开发、演示和自动化测试稳定。
- 可通过环境变量切换 DeepSeek。
- AI 只读取有权访问的文本培训资料。
- 题目草稿必须管理员审核。
- 不实现医疗 AI。

### 2.6 文件存储

当前采用本地文件存储并保留 `FileStorageService` 抽象。对象存储作为未来扩展，不作为轻量版强制依赖。

### 2.7 安全

- Spring Security + JWT。
- BCrypt 密码哈希。
- 动态加载最新账号状态和权限。
- 退出 Token 使用内存黑名单。
- 数据库密码、JWT Secret 和 AI Key 通过环境变量提供。

## 3. 结果

### 正面影响

- 交付范围与当前实现一致。
- 删除无前端闭环的大型业务域，降低联调和测试成本。
- 培训功能可以形成完整演示主线。
- Mock/DeepSeek 双模式兼顾稳定性和实际 AI 展示。
- 28 表结构更容易维护和最终建模。

### 负面影响

- `develop` 与 `lite_develop` 长期分叉，不能直接整体合并。
- 三个前端框架并存，依赖和交互规范需要额外维护。
- 内存 Token 黑名单不适合多实例生产环境。
- 本地文件存储需要部署方单独保证持久化和备份。

## 4. 被否决方案

### 继续开发完整版

否决原因：订单、医生和健康模块增加大量前端、联调、数据权限和测试工作，不能保证当前周期内形成稳定交付。

### 改为微服务

否决原因：团队和部署环境不需要，复杂度明显高于收益。

### 所有前端立即统一为 React 或 Vue

否决原因：会引入大规模重写，当前已有工程可运行，收益不足。

### AI 只保留真实 DeepSeek

否决原因：外部网络、Key、费用和响应不确定性会影响测试和演示。

### AI 草稿自动发布

否决原因：缺少人工审核会带来内容质量和培训安全风险。

## 5. 实施约束

- 轻量版 PR 以 `lite_develop` 为目标。
- 不重新引入 `care` 和 `doctor` 模块。
- 所有现行文档只描述两角色培训主线。
- 历史 Changelog 和 Test Log 保留完整版过程，不回写。
- 表结构变化同步更新 SQL、数据字典、设计、API 和测试。
- 需要改变本 ADR 时新增 ADR，不直接删除决策历史。
