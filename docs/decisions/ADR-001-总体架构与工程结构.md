# ADR-001 总体架构与工程结构

日期：2026-07-09

状态：已接受，T-011封板

## 背景

CareNexus 颐联需要在较短实训周期内完成护理培训主线和三端 MVP 闭环。需求基线 v1.0 已确认，T-010 详细测试用例已完成。T-011 需要确定架构和工程骨架，避免后续编码阶段大规模返工。

## 决策

采用以下架构：

- 前后端分离。
- 后端为一个 Spring Boot 2.7.x 应用，Java 8，模块化单体，单体部署。
- 前端为两个 Vue 3 + Vite 工程：`admin-web` 和 `mobile-web`。
- 四个逻辑业务端通过角色、菜单、路由、模块和接口边界区分，不按前端工程数量硬拆。
- `auth` 只负责当前用户解析、Token、RBAC 功能权限和账号状态校验。
- `admin` 只负责用户、角色、权限、账号和基础字典，不直接拥有所有业务表。
- `training` 负责培训资源、学习、考核和 AI 草稿审核。
- `care` 负责服务项目、地址、护理订单、分配、执行、评价和投诉。
- `doctor` 负责医生老人授权和健康管理。
- 医生老人授权由 `doctor` 模块通过 `DoctorAuthorizationService` 校验。
- 家属老人绑定和护工本人订单范围由 `care` 模块通过 `FamilyElderAccessService`、`CareOrderAccessService` 校验。
- 领域数据权限在业务模块 Service 层完成，`auth` 不跨领域查询医生授权、家属绑定或护理订单表。
- `audit` 负责操作日志。
- 业务模块之间只能通过 Service 接口协作，禁止跨模块直接访问其他模块 Mapper 或数据表。
- MySQL 8.0 作为主数据库。
- Redis 仅在明确需要登录状态、Token 黑名单、验证码或临时缓存时使用。
- 文件存储通过 `FileStorageService` 抽象，MVP 本地存储。
- AI 通过 `AiTrainingService` 独立适配层接入，支持 Mock 和真实实现切换。
- 医生端不实现 AI 诊断、AI 处方或自动医疗决策。

## 备选方案

### 方案A：单 Spring Boot 工程，按包划分模块

优点：

- 开发、运行和排错最简单。
- 适合 MVP 周期。
- 便于统一权限和事务。

缺点：

- 模块边界主要依靠代码规范和包结构约束。

### 方案B：Maven 多模块，单体部署

优点：

- 模块边界更清晰。

缺点：

- 配置和构建更复杂。
- 当前周期内容易增加联调成本。

### 方案C：微服务

优点：

- 长期扩展性强。

缺点：

- 明显超出 MVP。
- 部署、鉴权、数据一致性和联调成本过高。

## 结果

本项目选择方案A。

工程骨架将优先保证可构建、可启动、可演示，并保留后续扩展点。业务功能实现必须等待架构和数据库设计通过审核后按后续任务开展。

## 影响

- 后端代码集中在 `backend`。
- 前端代码集中在 `frontend/admin-web` 和 `frontend/mobile-web`。
- 数据库脚本集中在 `database`。
- 后续业务编码必须遵守模块边界、权限边界和 AI 边界。
- 综合管理端是前端入口，不代表 `admin` 模块直接拥有培训、订单、医生健康等业务表。
